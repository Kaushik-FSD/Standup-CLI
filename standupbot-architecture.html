Portfolio Project 03

StandupBot

CLI developer tool with AI-generated standups. Architecture reference & build guide.

01 / overview
What is StandupBot?
A personal developer productivity CLI. Log what you work on during the day in one command. Before standup, generate a clean AI-formatted summary in two seconds. End of week, get a full progress report.
Core Problem
Standup amnesia
Developers forget what they actually worked on. Brain goes blank. Logging takes 5 seconds and AI does the formatting.
Core Loop
Log → Generate
Log entries during the day. Run generate before standup. Clean formatted output printed straight to your terminal.
Portfolio Signal
CLI + Backend + AI
Rare combination. Most portfolio projects are web UIs. A polished CLI tool signals developer-tooling thinking.
AI Provider
Gemini 2.0 Flash
Free tier for development. Provider-agnostic abstraction layer — swap to Claude for production demo.
Scope
3 core features
Log, Generate, Weekly summary. Plus Config and Init. Tightly scoped to finish in 3–4 weeks.
Architecture Style
CLI → HTTP → Fastify
Commander.js CLI calls a local Fastify server. Clean separation — CLI is just the interface, backend does all the work.
02 / architecture
System Architecture
Three distinct layers. The CLI is a thin Commander.js interface. The Fastify backend is the brain — it handles auth, storage, and AI calls. Infrastructure is Docker-managed PostgreSQL and Redis.
Layer 1 — CLI Interface
CLI (Commander.js)
standup <command> [args] [flags]
Thin terminal interface. Parses commands, reads local config, sends HTTP requests to the local Fastify server. No business logic here.
Local Config
~/.standupbot/config.json
Stores user name, timezone, API server port, AI provider preference.
↓
Layer 2 — Fastify Backend (localhost)
Auth Module
/api/auth
API key based. User has a local API key stored in config. Every CLI request passes it as a header.
Logs Module
/api/logs
POST new log entry. GET logs for date range. DELETE individual entry. Core CRUD module.
Generate Module
/api/generate
Fetches recent logs, calls AI provider, returns formatted standup or weekly summary.
↓
Layer 3 — Infrastructure (Docker)
PostgreSQL
Persistent Log Storage
Stores all log entries with timestamps. Users table for API key auth. The permanent record — never wiped.
Redis
Cache + Rate Limiting
Caches today's logs for fast generate calls. Also handles generate rate limiting to avoid expensive AI calls on hammering.
AI Provider
Gemini 2.0 Flash (dev)
External API call. Abstracted behind AIProvider interface — can swap to Claude Sonnet for production without changing business logic.
03 / cli commands
CLI Interface
Built with Commander.js. Every command sends an HTTP request to the local Fastify server running on a configured port. The CLI is installable globally via npm install -g.
Command Flags Behaviour
standup log "<entry>" — POST /api/logs. Saves entry with current timestamp, date, and user ID. Takes 5 seconds. Core daily loop.
standup generate — GET /api/generate. Fetches logs for today + yesterday, sends to AI, returns formatted standup printed to terminal.
standup weekly — GET /api/generate/weekly. Fetches full week's logs (Mon–today), AI generates a progress summary. Used for weekly syncs.
standup history --days N GET /api/logs?days=N. Prints raw log entries for the past N days (default 7). No AI involved — raw record.
standup delete <id> — DELETE /api/logs/:id. Removes a specific log entry by ID. IDs shown in standup history output.
standup config --set key=value Reads or sets local config values: name, timezone, port, ai_provider. Stored in ~/.standupbot/config.json.
standup init — First-time setup wizard. Prompts for name and timezone. Generates local API key. Starts Fastify server.
04 / api modules
Fastify Backend Modules
Three route modules registered as Fastify plugins. Each module owns its routes, schema validation (Zod), and service layer. Prisma handles all database access.
auth Core
  • POST/api/auth/init
  • GET/api/auth/verify
Deps: PostgreSQL (users table), API key generation
Notes: API key stored in local config. Passed as x-api-key header on every request. init creates user + key on first setup.
logs Core
  • POST/api/logs
  • GET/api/logs?date=&days=
  • DEL/api/logs/:id
Deps: PostgreSQL (logs table), Redis (today's log cache)
Notes: POST invalidates Redis cache. GET checks Redis first, falls back to DB. Returns entries ordered by created_at.
generate AI
  • GET/api/generate
  • GET/api/generate/weekly
Deps: PostgreSQL, Redis (rate limit), AI Provider service
Notes: Rate limited to 10 calls/hour via Redis. Fetches logs, builds prompt, calls AI, returns formatted string. Cached for 5min per date.

07 / ai layer
AI Provider Abstraction
All AI logic lives in src/services/ai/. The interface is provider-agnostic. Dev uses Gemini 2.0 Flash (free tier). Production swap to Claude Sonnet by changing one env var.
Provider Interface
interface AIProvider {
generateStandup(logs: LogEntry[], context: StandupContext): Promise<string>
generateWeeklySummary(logs: LogEntry[]): Promise<string>
}
class GeminiProvider implements AIProvider { ... }
class ClaudeProvider implements AIProvider { ... }
// Resolved at startup via AI_PROVIDER env var
const provider = resolveProvider(process.env.AI_PROVIDER)
Prompt Design — Daily Standup
You are a developer standup assistant. Given these raw log entries from a developer, generate a clean standup summary. LOGS (yesterday): {yesterdayLogs} LOGS (today so far): {todayLogs} Format the response EXACTLY like this: Yesterday: [summarise what was done, concise] Today: [infer what's next based on incomplete/ongoing items] Blockers: [list any items marked as stuck, or "None"] Rules: - Be concise. One line per section unless multiple distinct items. - Don't repeat the same item in Yesterday and Today. - Infer "Today" intelligently from unresolved items in logs. - Never add commentary or preamble. Return only the formatted standup.
08 / tech stack
Technology Choices
Every choice made for a reason. No over-engineering. Same Fastify + PostgreSQL + Redis + Prisma 7 foundation as InfraCore.
CLI Framework
Commander.js
The standard for Node CLI tools. Simple, minimal, great argument/flag parsing. No overhead.
Backend
Fastify + TypeScript
Same stack as InfraCore. You know it. Fast, typed, plugin-based. No reason to switch.
ORM
Prisma 7
Same as InfraCore. You've already worked through Prisma 7's quirks. Consistent across projects.
Database
PostgreSQL
Structured log data. Date range queries are trivial SQL. MongoDB would be wrong here.
Cache
Redis
Cache today's logs for fast generates. Rate limit AI calls. TTL-based expiry built in.
AI (Dev)
Gemini 2.0 Flash
Free tier. 250 req/day. More than enough for development. Swap to Claude for production.
Validation
Zod
Request body and config validation. Pairs with Fastify's schema system. Consistent with InfraCore.
Infrastructure
Docker Compose
PostgreSQL + Redis in containers. One command to spin up. Same pattern as InfraCore.

10 / env config
Environment Variables
Server-side env vars only. The CLI reads from ~/.standupbot/config.json, not .env.
Variable Example Description
DATABASE_URL postgresql://user:pass@localhost:5432/standupbot Prisma database connection string. Points to Docker Postgres.
REDIS_URL redis://localhost:6379 Redis connection string for cache and rate limiting.
PORT 3001 Fastify server port. Must match CLI config port.
AI_PROVIDER gemini Which AI provider to use. Values: gemini | claude. Resolves the provider class.
GEMINI_API_KEY AIzaSy... Google AI Studio API key. Used when AI_PROVIDER=gemini.
ANTHROPIC_API_KEY sk-ant-... Anthropic API key. Used when AI_PROVIDER=claude. Leave blank in dev.
NODE_ENV development Controls logging verbosity and error detail in responses.
